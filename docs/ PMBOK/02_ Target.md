# プロジェクト目標
## ステークホルダーのニーズ
- 現行システムのデータ（ファイル、DB）は変更せずに使用
## 成果物
- FrontWeb：WebアプリケーションReactJS
- API Server:NodeJS+GraphQL(Appolo)
- 格納データ（MariaDB）
- Dockerイメージ(Web,API)
## スペック
### 機能要件
- すでに格納、存在するコミックスのデータを一覧、詳細、１冊ずつの本として表示できるもの
- ログイン認証
- 新着一覧
- タイトル中の一覧
- １冊の詳細情報（購入）
- ビューワー
- お気に入り追加・削除
### 非機能要件
- 一覧表の表示は５秒以内
- コミックのページ表示は１秒以内
- 格納するデータの制限はハードウェアの許す限り 
- 今回は対象外
  * 同時接続500ユーザー
  * レスポンス2秒以内
  * 稼働率99.9%
  * スマホ対応
  * SSO対応
### 対応範囲
- 含む：
  * フロントエンド
  * バックエンド
  * DB
  * 実行環境(AWS)構築
- 含まない：
  * MAツール連携
  * AI分析機能
  * 多言語対応

## 絶対条件・相対条件
- 現行システムのデータ（ファイル、DB）は変更せずに使用
- ビューワーはHTMLで構成するWebアプリケーションとする
## 除外項目
- データの投入、更新等は行わない
## 完了・成功の判断基準
- コミックのビューワーとしてストレスなく使えること

## 前提条件
- プロジェクトの実施にあたり、ほぼ確実に決まっていること
- プロジェクト実施を引き受ける上で、満たされるべきこと
+ 利用ユーザー数
  - 同時利用ユーザー数は最大10名以下とする
+ 顧客側リソース
  - 顧客側で要件確認担当者を週2回アサインできる
+ 既存システム連携
  - 既存システムのデータ登録は利用可能である
+ 開発体制
  - 開発メンバー5名をプロジェクト期間中固定で確保できる
+ 技術制約
  - AWS利用が社内承認済みである

#### 粒度として適切な理由
これらはすべて：
* スケジュール
* コスト
* 実現性
* リスク

に直接効きます。
