# 受託開発にAIを導入するための準備

## AI導入の手始め
- 受託開発のチームにこれらを強要することは難しいので、PMとしての自分が上記を導入し点数化するのはどうか？
- それをさらにチーム内に評価を共有するのはどうか？

重要なのは、
- 「評価制度」
ではなく、
- 「開発を楽にする仕組み」

やりたいのは、
- AI活用の成熟度可視化
- ただし「人評価」に直結させると危険


### Step1「品質改善支援」として導入
- NG AI使ってください
- OK 手戻り減らすために、AIで確認観点を増やしたい

### Step3 「チーム健康指標」にする
案件内で作成されたいろんなドキュメントをAIレビューしたかを数値表示する
- 基本設計書　レビュー済み　指摘事項　修正
- 詳細設計書　レビュー済み　指摘事項　修正
- テスト設計書　レビュー済み　指摘事項　修正
* AI利用でレビュー通過が速くなる

### Step4 AIをDoDへ組み込む
Definition of Done
* AIで要件確認済み
* AIで異常系観点抽出済み
* AIレビュー結果添付済み

#### 「確認漏れ防止」を行う

| LV | 状態 |

| AI活用 | 改善 |
|---|---|
| テスト生成 | 障害減少 |
| 要件レビュー | 手戻り減少 |
| PR要約 | レビュー時間減少 |
| 設計レビュー | 差し戻し減少 |


# AIの導入前の前提条件
## AI導入以前に基本的な考え方
- 抜け漏れをAIで指摘させる
    - 納品物のKPI化
    - 定義書の内容
    - 設計書の内容
    - テストの内容
### 人が判断し、AIが整理・草案化・網羅性チェックをする

## AI導入以前にも考える必要のあること
- タスクの小口化（標準化＆テンプレート作成）
- テストの小口化（標準化＆テンプレート作成）

# AIの導入
基本は人間だが、抜け漏れなどの指摘をさせる
| 工程 | AI活用 |
|---|---|
| 要件定義 | 曖昧点・確認事項抽出 |
| 基本設計 | 方式比較・リスク抽出 |
| 詳細設計 | API/DB/画面項目レビュー |
| 実装 | コード生成・リファクタ |
| テスト | テスト観点・項目作成 |
| レビュー | PR説明・影響範囲整理 |

## AIの導入をKPI化させるなら
| 目的 | KPI |
|---|---|
| 要件漏れ削減 | 仕様変更件数、手戻り件数 |
| 設計品質向上 | レビュー指摘数、設計起因バグ |
| テスト品質向上 | テスト観点数、障害流出件数 |
| 工数削減 | 要件整理時間、テスト作成時間 |
| 定着 | AIテンプレ利用率、PR記載率 |

# AI導入の目的
AIを単なる実装補助ではなく、要件定義・設計・テストの品質を標準化する仕組みとして導入したいと考えています。
特に受託開発では、要件の曖昧さやテスト観点のばらつきが手戻りの原因になりやすいため、AIで確認事項・リスク・テスト観点を自動抽出し、人が判断する運用に変えたいです。

本当の狙いは実装するところはほぼAIで均一化できるので、ささる提案を作成するために時間を使いたい

# 最初に作るべきもの
まずはこの3つです。
1. 要件定義AIチェックリスト
2. 設計レビューAIプロンプト
3. テスト項目生成テンプレート
