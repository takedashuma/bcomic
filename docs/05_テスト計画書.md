## 1. はじめに

### 1.1 目的

本書は zcomic-next システムの品質を担保するためのテスト方針・範囲・工程・体制・スケジュールを定義する。テスト項目書（ZCN-TC-001）と合わせて使用する。

### 1.2 適用範囲

  - apps/api（Apollo Server + Prisma + Express）の機能テスト

  - apps/web（Vite + React）の画面テスト

  - 既存MariaDB（tb_bok / tb_bkm / tb_usr）との結合

  - ファイルシステム（COMIC_ROOT）との結合

  - Docker Compose による統合動作

### 1.3 適用範囲外

  - 既存PHP版の動作確認（並行稼働の動作確認は最低限のみ）

  - MariaDB 自体のチューニング・負荷テスト

  - ネットワーク機器・ファイアウォール設定の検証

## 2. テスト方針

### 2.1 テストレベル

|              |                         |          |             |
| ------------ | ----------------------- | -------- | ----------- |
| **テストレベル**   | **目的**                  | **担当**   | **実施タイミング** |
| 単体テスト (UT)   | 個別関数・コンポーネントが仕様通り動くか    | 開発者      | コーディング中     |
| 結合テスト (IT)   | API ⇔ DB ⇔ ファイルシステム の連携 | 開発者      | 結合フェーズ      |
| システムテスト (ST) | E2E で全画面・全シナリオ          | QA / 開発者 | リリース前       |
| 受入テスト (UAT)  | 本番運用環境で利用者視点            | 発注者・社員   | リリース直前      |

### 2.2 アプローチ

  - API: GraphQL クエリ・REST エンドポイントを curl / GraphQL Playground で個別確認

  - Web: 実ブラウザ（Chrome / Safari）でシナリオを実施

  - DB: SQLクライアント（mysql コマンド / DBeaver）で前提データ作成・結果確認

  - 自動化は最小限。受入段階での回帰確認は手動チェックリスト

### 2.3 テストレベル間の関係

\[単体テスト\] 各 resolver / 各 React コンポーネント

↓

\[結合テスト\] GraphQL endpoint → Prisma → MariaDB

REST /img → fs → sharp → ストリーム

↓

\[システムテスト\] 6画面の操作フロー全体

並行稼働での共有テーブル整合性

↓

\[受入テスト\] 実コミックデータでの社内検証

## 4. テスト環境

### 4.1 環境構成

|         |         |                               |                  |                        |
| ------- | ------- | ----------------------------- | ---------------- | ---------------------- |
| **環境**  | **用途**  | **DB**                        | **COMIC_ROOT**  | **備考**                 |
| ローカル開発  | 個別動作確認  | ローカル MariaDB or DEV-DB        | テスト用ミニフォルダ       | vite dev + tsx watch   |
| DEV     | 結合テスト   | 192.168.40.99（DEV instance推奨） | 実ファイル（read-only） | Docker Compose         |
| STAGING | システムテスト | 実DB（読書込み許可）                   | 実ファイル            | 本番同等構成                 |
| 本番      | 受入・運用   | 実DB                           | 実ファイル            | Synology Reverse Proxy |

### 4.2 必要ハードウェア

  - API/Webコンテナ実行ホスト: メモリ4GB以上、Docker Desktop 4.x

  - MariaDBホスト: 既存 192.168.40.99 を流用

  - クライアント: Chrome 最新、Safari 最新、社内標準PC

### 4.3 ソフトウェア前提

|                |                     |
| -------------- | ------------------- |
| **項目**         | **バージョン**           |
| Node.js        | 20.x                |
| Docker Compose | v2                  |
| MariaDB        | 10.11.6（既存）         |
| ブラウザ           | Chrome 最新、Safari 最新 |

### 4.4 テストデータ

  - ユーザー: テスト用に test_admin / test_user_a / test_user_b の3アカウントを tb_usr に作成

  - コミック: 既存データをそのまま利用（最低5作品・各3〜10巻）

  - お気入: テスト前にユーザーごとに 2件を tb_bkm に登録

  - 閲覧進捗: 必要に応じて tb_red に登録

## 5. 体制と役割

### 5.1 体制図

プロジェクトマネージャ（1名）

│

├─ 開発リーダー（1名）

│ └─ 開発担当（2名） : UT実施、IT実施

│

├─ テストリーダー（1名）: ST計画・指揮

│ └─ テスト担当（1〜2名） : ST実施、不具合管理

│

└─ 業務担当（社員）: UAT実施

### 5.2 役割分担

|          |                          |
| -------- | ------------------------ |
| **役割**   | **責任範囲**                 |
| PM       | 全体スケジュール、リスク管理、報告        |
| 開発リーダー   | UT/IT 計画作成、技術的判断、コードレビュー |
| 開発担当     | 実装・UT・IT実施               |
| テストリーダー  | ST計画策定、テストケース作成、不具合トリアージ |
| テスト担当    | ST実施、不具合チケット起票、回帰確認      |
| 業務担当（社員） | UAT実施、受入判定               |

### 5.3 不具合管理

  - 不具合は GitHub Issues（または社内チケットツール）で管理

  - 重要度: Critical / High / Medium / Low の4段階

  - ステータス: Open / In Progress / Fixed / Verified / Closed

  - Critical/High は当日対応、Medium は翌営業日まで、Low は次回リリース前

### 6.1 全体スケジュール（目安）

|              |        |               |
| ------------ | ------ | ------------- |

  - M2: IT完了 — 全GraphQL/RESTが手動疎通OK

  - M3: ST完了 — 6画面のシナリオ通り動作確認

  - M4: UAT合格 — 業務担当者の合意取得

  - M5: 本番リリース — 並行稼働開始

## 7. 完了基準

### 7.1 単体テスト完了基準

  - 全 GraphQL resolver の正常系・主要異常系をローカルで確認

  - 全 React 画面の表示・操作をローカル開発サーバで確認

  - Critical / High 不具合 0件

### 7.2 結合テスト完了基準

  - テスト項目書の IT 項目すべて消化

  - Pass率 100%（または許容不具合のみ）

  - 既存PHP版との並行整合性確認 OK

### 7.3 システムテスト完了基準

  - テスト項目書の ST 項目すべて消化

  - Pass率 95%以上

  - Critical / High 不具合 0件、Medium 5件以下

  - 性能要件達成

  - セキュリティ要件達成（未認証アクセス、パストラバーサル）

### 7.4 受入テスト完了基準

  - 業務担当者による全機能の動作確認 OK

  - 発注者からの受入合意書取得

### 7.5 中止判断基準

  - Critical 不具合が連続3件以上、修正に2日以上必要な場合は工程を中断し再計画

  - DB / ファイルシステムの破損リスクが発覚した場合は即中断

## 8. リスクと対策

|                                 |         |          |                                                    |
| ------------------------------- | ------- | -------- | -------------------------------------------------- |
| **リスク**                         | **影響度** | **発生確率** | **対策**                                             |
| 既存DBへの接続権限不足                    | 高       | 中        | 事前に管理者と接続情報・権限を合意。SELECT/UPDATE/INSERT の権限を限定的に付与。 |
| 並行稼働で tb_bkm のID衝突             | 中       | 低        | MAX(id)+1 採番をトランザクション化。レアケース時は手動修正。                |
| 平文パスワードを bcrypt 化したことでPHP版が認証失敗 | 高       | 中        | PHP側にbcrypt対応コードを追加するか、平文に戻すマイグレ手順を準備。             |
| COMIC_ROOT のNFS切断              | 中       | 低        | API側でファイル取得失敗時の404返却。再マウント手順を文書化。                  |
| MariaDB のリモート接続不可               | 高       | 中        | bind-address と GRANT を事前検証。トラブルシュート手順を文書化。         |
| ARM64 Mac での Prisma エンジン問題      | 中       | 高        | node:20-bookworm-slim + binaryTargets で回避済み。       |
| Vite/Express のバージョン互換性          | 中       | 中        | package.json でバージョンピン、CI で再現確認。                    |
| TB級画像での性能劣化                     | 中       | 中        | サムネのキャッシュ追加検討。fs.readdir の LRU 追加検討。               |
