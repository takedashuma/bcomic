## 1. はじめに

### 1.1 目的

本書は zcomic-next（以下「本システム」）の基本設計を示す。本システムは既存のPHP製コミックビューワー（以下「現行システム」）の機能のうち、社内ユーザー向けの作品閲覧機能に絞ったうえで、Node.js + GraphQL + React 構成で再構築する。

### 1.2 背景

  - 現行システムはPHP 8.0 + MariaDB 10.11 で構築されており、保守性・拡張性の観点で限界がある。

  - 画面構成にポイント購入や成人モードなど業務上不要な機能が混在しており、社内利用には冗長。

  - 既存DB（tb_bok / tb_bkm / tb_usr）には10年以上の蓄積データがあり、移行コストが大きい。

  - コミック画像はファイルシステム（/Volumes/public/Comic/Renamed、TB級）に保管されており、再配置は非現実的。

### 1.3 スコープ

#### 1.3.1 対象範囲

  - 会員ユーザー向けのコミック作品閲覧Webアプリケーション。

  - 既存MariaDBの tb_bok / tb_bkm / tb_usr の参照および一部更新。

  - 既存ファイルシステム配下のコミック画像配信。

  - Docker Compose による配信構成（API/Webコンテナ + 既存外部MariaDB）。

#### 1.3.2 対象外

  - ポイント機能・購入機能（tb_pth）

  - 成人モード、無料モード、お勧め切替などのモード分け

  - コミックデータの新規登録・編集（現行PHP版が継続担当）

  - 管理者画面（tb_adm 活用）

## 2. システム概要

### 2.1 全体像

本システムは社内ネットワーク上に Docker Compose でデプロイされ、Webブラウザを通じて社員が利用する。データは既存の社内MariaDB（192.168.40.99）に直接接続して取得し、コミック画像は同サーバ上のファイルシステムからストリーミング配信する。

### 2.2 配置構成

  - 既存PHP版 / 既存MariaDB / 既存COMIC_ROOT はすべて 192.168.40.99 上に存在。

  - 本システム（Docker Compose）も同じ 192.168.40.99 上に同居させる。これにより既存ファイルシステムへ bind mount でアクセス可能。

  - ホスト1台に集約されるため、API・Web・DB・FS の間はすべてローカル通信となり、レイテンシ最小。

### 2.3 アクセス経路

|            |                                                            |                                    |
| ---------- | ---------------------------------------------------------- | ---------------------------------- |
| **経路**     | **URL**                                                    | **用途**                             |
| 社内LAN（推奨）  | http://192.168.40.99:8080/                                 | 通常利用                               |
| 外部（DDNS経由） | http://dirs.synology.me:8080/ or https://dirs.synology.me/ | 社外からの参照（Synology Reverse Proxy 経由） |
| API直アクセス   | http://192.168.40.99:4000/                                 | Swagger UI (/docs)、運用デバッグ          |

フロントエンドのAPI呼び出しは「ページを読み込んだのと同じオリジン」に対する相対URL方式を採用する。これにより VITE_API_URL 環境変数を空に設定するだけで LAN/WAN 両対応となり、ビルド成果物に固定IPが焼き込まれない。

### 3.4 機能仕様

#### 3.4.1 F-01 ログイン

入力

  - メールアドレス（tb_usr.usr_vch0）

  - パスワード（tb_usr.usr_vch1）

処理

## 1. メールアドレスで tb_usr を検索し、論理削除されていないレコードを取得。

## 2. パスワード照合。保存形式が平文の場合は文字列一致後、bcryptハッシュへ自動マイグレーション。bcryptハッシュの場合は bcrypt.compare で検証。

## 3. 認証成功時：tb_usr.usr_lgday を現在時刻で更新、JWTを発行して httpOnly Cookie に格納。

## 4. 認証失敗時：エラー INVALID_CREDENTIALS。

出力

  - User { id, email, nickname } を返却。

  - Cookie zc_token=\<JWT\>; HttpOnly; SameSite=Lax; Max-Age=7d を設定。

#### 3.4.2 F-03 新着順 一覧

対応SQL（仕様の根拠）

SELECT bok_vch6, bok_vch7,

MAX(bok_txt1), MAX(bok_mid) AS max_bok_mid,

MAX(bok_vch8) AS max_bok_vch8, COUNT(bok_vch8) AS cnt_bok_vch8

FROM tb_bok

WHERE bok_dlday IS NULL

AND bok_vch9 = 'comic'

GROUP BY bok_vch6, bok_vch7

ORDER BY max_bok_mid DESC

LIMIT 24 OFFSET ?;

表示条件

  - tb_bok.bok_dlday IS NULL（論理削除除外）

  - tb_bok.bok_vch9 = 'comic'（コミック種別のみ）

集約

  - GROUP BY (bok_vch6, bok_vch7) — 日本語の著者・タイトル単位で作品化。

ソート

  - ORDER BY MAX(bok_mid) DESC — 各作品で最も新しい巻のIDが大きい順。

ページネーション

  - 1ページあたり24件。LIMIT/OFFSET。

検索

  - クエリ q が指定された場合、bok_vch2/3/6/7 のいずれかに部分一致。

出力フィールド

  - authorEn, titleEn（代表値：最新巻の英字名）

  - authorJa, titleJa（日本語名）

  - volumeCount（COUNT(\*)）

  - latestVolume（最新巻情報、サムネ用）

  - isFavorite（ログインユーザーのお気入登録有無）

#### 3.4.3 F-04 お気入 一覧

F-03 と同条件 + 以下を追加。

  - ログインユーザーが tb_bkm に登録済みの (bkm_vch6, bkm_vch7) に限定。

  - bkm_dlday IS NULL（論理削除除外）。

#### 3.4.4 F-05 お気入トグル

## 5. 対象作品の英字キーから tb_bok を引き、日本語名 (authorJa, titleJa) を解決。

## 6. 既存の tb_bkm レコード（bkm_uid + 日本語名一致 + bkm_dlday IS NULL）を検索。

## 7. 既存があれば bkm_dlday を現在時刻で更新（論理削除）。

## 8. 無ければ tb_bkm の最大IDから連番採番して新規登録。

## 9. 返却値：登録後のお気入状態（true=登録 / false=解除）。

#### 3.4.5 F-06 詳細

URLパラメータ (authorEn, titleEn) で受け取り、tb_bok を英字でいずれか1巻引いて日本語名を解決。同じ日本語名の全巻を bok_vch4 昇順で表示する。

#### 3.4.6 F-07 ビューワー

  - 対象巻のフォルダ（bok_txt1）配下の画像ファイルをファイル名昇順（自然順ソート）で1枚ずつ表示。

  - ページ送り：右クリック/→キー/右→左スワイプで次ページ、左クリック/←キーで前ページ。

  - ページ番号スライダーで任意ページにジャンプ可。

  - 最終閲覧ページを tb_red に保存（debounce 1.5秒）。

  - 再訪時は前回の最終ページから開始。

  - 隣接ページ（前後1枚）を Image オブジェクトで先読み。

#### 3.4.7 F-08 パスワード変更

## 10. 現パスワードを検証（平文の場合はマイグレーション処理含む）。

## 11. 新パスワード長 \< 6 文字なら PASSWORD_TOO_SHORT。

## 12. bcryptハッシュ化して tb_usr.usr_vch1 に保存。

#### 3.4.8 F-09 / F-10 画像配信

  - F-09 サムネ：GET /img/cover/:volumeId?w=400 — sharp で幅400px JPEG にリサイズ。

  - F-10 本ページ：GET /img/page/:volumeId/:pageIndex — オリジナル画像を Range request 対応で配信。

  - いずれも JWT cookie 認証を要求（401 を返す）。

  - パス解決時に COMIC_ROOT 配下であることを resolve 後に検証（パストラバーサル対策）。

## 4. 非機能要件

### 4.1 性能

|           |             |                 |
| --------- | ----------- | --------------- |
| **項目**    | **目標値**     | **備考**          |
| 新着順 一覧表示  | 1.5秒以内（24件） | サムネ画像取得は除く      |
| サムネ初回取得   | 500ms以内/枚   | sharpで都度リサイズ    |
| ビューワー次ページ | 200ms以内     | 先読みヒット時は瞬時      |
| ログイン処理    | 1秒以内        | 平文→bcrypt変換時を除く |
| 同時接続数     | 50ユーザー      | 社内利用想定          |

### 4.2 可用性

  - 業務時間中（平日 9:00-19:00）の稼働率 99%。

  - 計画停止：月1回程度、業務時間外に実施可。

  - 外部MariaDB停止時はアプリも停止する（HA構成は対象外）。

### 4.3 セキュリティ

  - 認証：JWT + httpOnly Cookie。SameSite=Lax。

  - パスワード：bcrypt（cost 10）で保存。平文保存は移行期のみ許容。

  - セッション：JWT有効期限 7日。

  - CORS：許可Originをホワイトリストで明示。

  - 通信：本番ではHTTPS化推奨（Synology DSM等のリバースプロキシで Let's Encrypt）。

  - ファイルアクセス：COMIC_ROOT 配下に限定（パストラバーサル対策）。

  - 画像配信は認証必須（401返却）。

### 4.4 運用

  - ログ：API側はstdout出力。Docker logsで参照。

  - デプロイ：docker compose up -d --build による全体更新。

  - バックアップ：既存MariaDBのバックアップ運用に依存。tb_red も同DB内のため同時バックアップ。

  - 監視：docker compose ps と /healthz エンドポイント。

### 4.5 拡張性

  - 追加画面はTanStack Router にroute定義を追加するのみ。

  - 追加テーブルは tb_red と同様に SQL でCREATE し、Prismaスキーマに@mapで追加。

  - 画像形式追加（WebP/AVIF）はsharp対応のみ。

## 6. 制約事項

### 6.1 技術制約

  - 既存DBのテーブル定義（tb_bok / tb_bkm / tb_usr）は変更不可。新テーブル tb_red のみ追加可。

  - 既存PHP版が稼働中。tb_bok への書き込みは本システムでは行わない。

  - コミック画像はファイルシステム保管のまま。DBへのBLOB移行は行わない。

  - MariaDB 10.11 の MySQL互換プロトコルで接続するため、Prisma の datasource provider は 'mysql' を指定。

  - Apple Silicon / Linux x86_64 双方で動作する必要があるため、Prisma の binaryTargets に linux-arm64-openssl-3.0.x を含める。

### 6.2 運用制約

  - 外部MariaDBへのリモート接続が必要。bind-address = 0.0.0.0、ユーザーHost = '%' or サブネット指定。

  - 外部公開する場合は Synology DSM のリバースプロキシ等で TLS 終端する。

  - COMIC_ROOT ボリュームは macOS Docker Desktop で File sharing 許可が必要。

### 6.3 移行制約

  - 平文パスワードは初回ログイン時に bcrypt へ自動移行。

  - 既存お気入レコードで日本語名が NULL の場合、SQL補修が必要。

  - ユーザー登録機能は本システムでは実装しない。管理者が tb_usr に直接INSERTする運用。
